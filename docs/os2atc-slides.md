# 背景：AI 芯片 ISS 仿真的性能困境

```
┌─────────────────────────────────────────────────────┐
│  AI 芯片设计验证流程                                  │
│                                                     │
│  RTL 仿真 ──→ ISS 仿真 ──→ FPGA 验证 ──→ 流片      │
│  (最慢)      (瓶颈↑)      (受限)       (成本最高)    │

```

# 什么是动态二进制翻译？

```
              翻译流水线 (Translation Pipeline)

  ┌──────────┐    ┌────────┐    ┌──────────┐    ┌──────────┐
  │ Guest ELF│───→│ 前端   │───→│  TCG IR  │───→│  优化器  │
  │ (RISC-V) │    │ 解码   │    │(中间表示)│    │ 常量折叠 │
  └──────────┘    └────────┘    └──────────┘    │ 拷贝传播 │
                                                └────┬─────┘
  ┌──────────┐    ┌────────┐    ┌──────────┐         │
  │ 执行结果 │◀───│ JIT    │◀───│  后端    │◀────────┘
  │          │    │ 执行   │    │ x86-64   │
  └──────────┘    └────────┘    │ 代码生成 │
                       ↑        └──────────┘
                  ┌────┴─────┐
                  │ TB 缓存  │  一次翻译，多次执行
                  └──────────┘
```

**关键优势**：JIT 编译 vs 解释执行 = **10~50x** 性能差距

# 翻译块（TB）生命周期

```
  ┌─────────────────────────────────────────────────────┐
  │  PC Hash ──→ Jump Cache (4096) ──→ 全局 Hash (32K)  │
  │                  │ 命中                │ 命中         │
  │                  ▼                    ▼              │
  │  ┌──────────────────────────────────────┐           │
  │  │         已缓存的 TB                   │◀─ 缓存    │
  │  └────────────┬─────────────────────────┘           │
  │               │ 未命中                               │
  │               ▼                                     │
  │  前端解码 → IR → 优化 → 后端 codegen → 缓存 TB      │
  │               │                                     │
  │               ▼                                     │
  │  执行 → goto_tb 直接链接 → 下一个 TB                 │
  └─────────────────────────────────────────────────────┘
```

**tcg-rs 额外优化**：`next_tb_hint` + `exit_target` 原子缓存（后面详述）

# 关键设计决策：tcg-rs vs QEMU

| 设计点 | tcg-rs (Rust) | QEMU (C) | 收益 |
|--------|--------------|----------|------|
| Opcode | 158 个统一多态 | ~250 个按类型拆分 | 减少 40% 分派 |
| IR 存储 | `Vec<Op>` 连续内存 | 链表 `QTAILQ` | 缓存友好 |
| 寄存器分配 | 约束驱动声明式 | per-opcode switch | 零分支通用 |
| Opcode 大小 | `#[repr(u8)]` 1B | `int` 4B | 4x 紧凑 |
| 常量查找 | `HashMap` O(1) | 线性扫描 O(n) | 大 TB 加速 |
| 解码器生成 | proc macro 编译期 | Python 脚本外部 | 无依赖 |
| 内存访问 | 直接 `[R14+addr]` | 软件 TLB 路径 | 指令数 1/5 |

---

# 代码示例：统一 Opcode + 约束分配器

```rust
// 统一多态 Opcode — 158 个 vs QEMU ~250 个
#[repr(u8)]
pub enum Opcode {
    Mov = 0, Add, Sub, Mul, Div, Rem,
    And, Or, Xor, Shl, Shr, Sar,
    // ... 类型信息在 Op::op_type，不在 opcode 里
}

// 约束驱动寄存器分配 — 零 per-opcode 分支
Opcode::Add => o1_i2_alias(REGS_ALL, REGS_ALL, REGS_ALL),
Opcode::Div => o1_i2_alias_fixed(RAX, REGS_ALL, REGS_ALL),
// 新增 opcode？加一行约束即可，分配器零改动
```

```
  QEMU 方式                    tcg-rs 方式
  ┌────────────┐              ┌────────────┐
  │ add_i32    │              │   Add      │
  │ add_i64    │  ──合并──→   │ (type=I64) │
  │ add_vec    │              │ (type=V128)│
  └────────────┘              └────────────┘
```

# 性能优化贡献分解

| 优化类别 | 估算贡献 | 关键技术 |
|---------|---------|---------|
| 执行循环 | ~8-10% | `next_tb_hint` + `exit_target` 缓存 |
| Guest 内存访问 | ~8-10% | 无 TLB，直接 `[R14+addr]` 寻址 |
| 数据结构 | ~3-5% | Vec 连续存储 + `#[repr(u8)]` 枚举 |
| 运行时并发 | ~2-3% | Lock-free TB + 无 mprotect 切换 |
| 编译管线 | ~2-3% | 单遍优化器 + Rust 零成本抽象 |
| 哈希 + 常量 | ~1-2% | 黄金比例哈希 + HashMap 去重 |
| **合计** | **~24-33%** | |

```
  贡献占比 (可视化)
  执行循环    ████████████████████  8-10%
  内存访问    ████████████████████  8-10%
  数据结构    ██████████            3-5%
  运行时并发  ██████                2-3%
  编译管线    ██████                2-3%
  哈希+常量   ████                  1-2%
```

# 优化案例 1：next_tb_hint 跳过 TB 查找

```
  QEMU 执行循环                    tcg-rs 执行循环
  ┌──────────────┐                ┌──────────────┐
  │ 执行 TB-A    │                │ 执行 TB-A    │
  └──────┬───────┘                └──────┬───────┘
         │ goto_tb 退出                   │ goto_tb 退出
         ▼                               ▼
  ┌──────────────┐                ┌──────────────┐
  │ jump cache   │                │ next_tb_hint │
  │ hash 查找    │                │ 直接复用目标  │
  └──────┬───────┘                └──────┬───────┘
         │ miss?                         │ 命中率极高
         ▼                               ▼
  ┌──────────────┐                ┌──────────────┐
  │ 全局 hash    │                │ 执行 TB-B    │
  │ 查找         │                │ (零查找开销) │
  └──────┬───────┘                └──────────────┘
         ▼
  ┌──────────────┐
  │ 执行 TB-B    │
  └──────────────┘
```

**效果**：紧密循环（如 dhrystone）中 hint 命中率极高，几乎跳过所有查找

# 优化案例 2：无软件 TLB 直接寻址

```
  QEMU (linux-user 模式仍走 TLB)        tcg-rs (直接寻址)

  mov rax, [guest_addr]                 mov rax, [R14 + guest_addr]
  ;; 内联 TLB 快速路径:                   ;; 仅 1 条指令！
  ;;   计算 TLB 索引      (2 条指令)
  ;;   加载 TLB 项         (1 条)
  ;;   比较页地址           (1 条)
  ;;   条件跳转慢速路径     (1 条)
  ;;   加载数据             (1 条)
  ;;   合计: 5-10 条指令
```

| 对比 | tcg-rs | QEMU |
|------|--------|------|
| 每次访存指令数 | 1-2 条 | 5-10 条 |
| TLB miss 处理 | 无 | helper 函数调用 |
| 适用场景 | linux-user | 全部模式 |

**代价**：仅限 linux-user 模式，不支持系统模式页表
